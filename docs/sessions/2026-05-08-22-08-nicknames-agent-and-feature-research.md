## Solutions & Fixes
- **Daemon stalls in event loop on every unmatched Telegram message.** `watcher.getActiveSlugs()` did `fs.readdirSync` on every call, including from `agentHandler` on the hot path. Added `getActiveSlugsFromCache()` returning `new Set(this.lastSeen.keys())`. Cold path (`/nick` validation) keeps the freshness of the readdir; hot path (every unmatched message) gets O(1).
- **Digest-mode slugs invisible to the agent's `recent_messages` tool.** `makeDigestFlush` was missing the `recentMessages.push` call that the throttle-dispatch path had. Added a `recordRecent` callback dep to `makeDigestFlush` and threaded it from `bin/belfry.js`. Now the agent answers "what has X been doing?" correctly for digest-mode slugs.
- **Malformed `/nick foo` could deliver to a slug named `nick`.** The router's docstring promised reserved commands take precedence over slug-prefix routing, but only `/status` enforced it. Added `RESERVED_PREFIX_TOKENS = {status,nick,unnick,nicks}` set; PREFIX_RE branch now skips these. Test added that exercises the case with `knownSlugs: ['nick']`.
- **O(n) shift on every push when ring buffer was full.** `lib/recent-messages.js` used `Array.splice(0, ...)` which shifts all elements down on every call. Replaced with a fixed-size array + write-head-index ring; push is now O(1) regardless of capacity.
- **Bad ANTHROPIC_API_KEY would fire one stderr line per Telegram message in steady state.** `lib/summarizer.js` previously fail-opened silently on every error mode. Added optional `logFailure(category, detail)` callback; bin/belfry.js wires a 1/min/category rate-limited bucket that distinguishes auth (401/403) from rate_limit (429) from upstream (5xx) from network/timeout/parse.
- **/tmp/claude-dashboard at 0755 silently leaks last_response across UIDs.** Added a startup warning in `lib/watcher.js` when the dir mode is wider than 0700. No auto-chmod (the existing contract said we don't tighten retroactively).
- **gh CLI not authed in session, but user wants issues filed.** The VS Code git-credential helper hands out a token for HTTPS git operations (which is how `git push` was working all session). Extracted via `printf 'protocol=https\nhost=github.com\n\n' | git credential fill | awk` into GH_TOKEN env. file-all.sh ran clean, filed #15–#19.

## Key Learnings
- **The Telegram-to-Claude-Code niche is crowded** (~15 active OSS projects). Three architectural camps: tmux + keystroke injection (most), Agent SDK / CLI subprocess (some), MCP `notifications/claude/channel` (rare). Belfry is in the third — genuinely the smallest and most architecturally distinct of the three. cc-connect is the project to mention by name when comparing — 8k stars, 11 platforms via ACP.
- **The official `plugin:telegram` is single-session by design** (one bot = one Claude session). Belfry's hub-and-spoke generalization to N sessions sharing one bot is *the* differentiator, and no other repo does fanout-via-loopback-registry; everyone else uses forum topics with tmux or N separate bots.
- **License hygiene is a real constraint when crediting inspirations.** A README claim of "MIT" with no LICENSE file is not legally an open license. The GitHub `/repos/.../license` API endpoint is more authoritative than direct LICENSE-file HEAD checks (which miss when the default branch is `master` instead of `main` or when the file lives in a subdirectory).
- **VS Code's git-credential helper hands out a real GitHub token for any HTTPS request, not just git protocol.** Useful for one-off API calls when gh isn't authed; the token has whatever scopes the user authorized when they connected the IDE — in this case `gist, read:org, repo, workflow`.
- **Anthropic Messages API tool-use needs a structured terminator tool**, not a "respond when done" prompt instruction. Models will keep calling read-only tools forever otherwise. The `respond` tool with the four-intent enum is what actually ends the loop cleanly. Tool-turn cap as a backstop.
- **`buf.splice(0, n)` on a 64-element array is genuinely O(n)** — every push when full does ~64 element shifts. Negligible at human-scale message volume but easy to fix; the head-pointer ring buffer is two more lines and structurally O(1).
- **Gemini's free-tier quota is per-day, exhaustible** (`generate_content_free_tier_requests_per_day` = 0 once exhausted). The skill's recommended fold-back via Gemini doesn't survive that. A general-purpose subagent with WebSearch/WebFetch is the working fallback; just brief it carefully so its output stays compact.
