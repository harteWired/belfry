## Solutions & Fixes
- **`kill -0 <pid>` false positive on recycled PIDs.** Real root cause of why today's outage couldn't self-heal on the next postattach. Fixed in two places: (1) doctor's `isSupervisorPid()` reads `/proc/PID/cmdline` and grep-matches the supervisor body's `LOGFILE="<path>"` marker; (2) launcher's idempotency check does the same with `grep -aq` on `/proc/$pid/cmdline`. PIDs may collide; cmdline strings of our specific script body don't.
- **SIGHUP cascade through controlling-terminal teardown.** Three-layer fix: `setsid` on the outer bash (new session, no controlling tty), `trap "" HUP` inside the bash (no-op handler), and `process.on('SIGHUP', () => log(...))` in node (ignored at the process level too). Even if one layer fails, the others hold.
- **Supervisor death is silent (no death-log entry).** Added an EXIT trap to the supervisor bash that writes `"$ts supervisor exit"` to `belfry-deaths.log` before exiting. Combined with the new doctor, you now have *two* observability paths: the deaths log records the event, and the doctor's "supervisor: fail" finding surfaces the current state.
- **Mismatched cmdline marker pattern on first try.** My first version of `isSupervisorPid()` checked for `LOGFILE=${LOGFILE}` (no quotes); the bash cmdline preserves the quoted form. Caught immediately on dry-run: doctor incorrectly diagnosed the live supervisor as "PID likely recycled." Patched the doctor + launcher to use `LOGFILE="${LOGFILE}"` (with quotes) — also more specific, less risk of false positives from env-var assignments elsewhere.
- **package.json edit clobbered by external process mid-edit.** When I tried to add `belfry-doctor` to the bin map, the Edit tool errored "File has not been read yet" — the harness had detected an external write to `package.json` (presumably an editor or a watcher) and invalidated its cache. Re-read the file, re-applied the edit, moved on. Cost: 2 tool calls. Worth noting because the working tree had other concurrent modifications too.
- **Test suite hangs at scale.** `npm test` ran for >60s and was killed. Known issue from the 2026-05-09 session — `node --test test/` hangs around the digest tests when 30+ files run together with subprocess-spawning brain tests. Workaround used: ran targeted files (`node --test test/poller.test.js test/voice.test.js test/composer.test.js`) — 59 tests, all green. No new tests added for the doctor (pure ops tooling exercised end-to-end by the restart + verification flow).
- **README modification mid-session shows the working tree isn't quiet.** Between commits, package.json went from `0.1.0` to `0.1.1` without my edit landing (the harness flagged it as "modified, either by the user or by a linter"). No harm — the value I wanted was already there. Logged for awareness; future Edit calls in this repo should expect concurrent writes.

## Key Learnings
- **`kill -0 <pid>` is not a liveness proof in any system where PIDs can be reused** — and they always can be on Linux. For any "is process X still alive" check beyond a trivial "did my immediate child die" wait(), you need a process-identity match: cmdline contents, controlling tty, parent PID, start time, or all of the above. The recycled-PID failure mode is rare enough to be invisible until it bites; today it bit for ~45 minutes before the user noticed.
- **`nohup` + bash's job control doesn't reliably preserve SIG_IGN for SIGHUP across the child chain.** `nohup` SIG_IGNs HUP for the bash, but Bash's job-control machinery (background jobs in their own pgid) can reset signal dispositions when spawning. `setsid` is the actually-reliable detach — it puts the process in a new session with no controlling terminal, so the kernel has nothing to SIGHUP it on terminal teardown. In any future "make this daemon survive terminal close" wiring, reach for `setsid` first.
- **Per-process unique markers in `/proc/PID/cmdline` are a load-bearing affordance on Linux** for process identity beyond PIDs. Any supervisor script that needs idempotency-by-cmdline should bake in a distinctive literal (a unique env-var assignment, a flag, a comment in the script body) early in the cmdline string, so a grep can disambiguate cheaply.
- **The devcontainer has no cron, no systemd, no crontab.** Realized this when looking for a system-level watchdog. The "make this self-healing in your container" answer is necessarily Claude-coupled (`/loop`) or out-of-container (host cron). For belfry's threat model this is fine — the supervisor + a /loop watchdog gives two independent defenses, and PRs to add cron to the container are scope creep.
- **Telegram-side outages went unobserved for ~45 min in today's case** because there's no heartbeat ping from the daemon to the chat. v0.1.1 doesn't add one — would have been scope creep — but it's a natural follow-up: a hourly "belfry up since <timestamp>" or a heartbeat that posts only when the doctor finds a state change.
- **Releases tagged on a feature branch are a fine pattern for pre-merge stabilization.** v0.1.0 / v0.1.1 sit on `feature/mcp-hub-spoke`; PR #10 will eventually merge them to `main`. Tags are immutable references — when PR #10 merges, the SHAs they point to are preserved in main's history (just deeper in the DAG). No need to re-tag after merge.
