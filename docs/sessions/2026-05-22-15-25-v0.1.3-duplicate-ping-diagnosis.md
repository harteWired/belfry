## Solutions & Fixes
- **Diagnosis only — no code changed.** The session was bounded to evidence-gathering + framing for the next session. v0.1.3 PingDedup IS working for the watchdog scenario it was designed for, per `belfry.log` at 21:03:58 / 21:11:26 / 21:21:28.
- Recommended follow-up shape (not implemented): suppress the next ready ping for a slug when `last_response` equals the most-recent outbound reply text just sent for that slug. Small LRU keyed by slug → `{ text, ts }`, with a short recency window (5-10s) so legitimate later turns still ping. Switch `PingDedup` from `last_response_at` equality to `last_response` text equality at the same time — fixes both the reply-tool double-send and the second-boundary drift in one structural change.

## Key Learnings
- **PingDedup as shipped only catches the static-content scenario** (`/loop` watchdog re-pinging the same `last_response` across consecutive Stop events with the same `last_response_at`). It does NOT catch "same content, fresh timestamp" — which is exactly what happens after a reply-tool call. The design implicitly assumed `last_response_at` was a content fingerprint; it's a write-time stamp.
- **Reply-tool + Stop-hook ping is a structural duplicate by design.** The reply tool emits a Telegram message immediately via `/send` → `sendOutbound`. The Stop hook then captures the same assistant text into the dashboard, the watcher fires `onUpdate`, and the composer emits a "ready" status ping carrying that same text. Two messages for one logical reply. Pre-v0.1.3 this was lost in the noise of the /loop amplification (5+ pings per content); post-v0.1.3 the 2:1 ratio is now the visible floor.
- **Hook attribution surprise:** belfry-hook.js shipped in the belfry repo isn't the one actively running here. `~/.claude/hooks/dashboard-status.js` (claudelike-bar's hook, v0.18.1+) is wired in `~/.claude/settings.json` and is what writes the dashboard JSON. The v0.1.3 matching patch was applied to that file at 21:58 UTC (file mtime confirms). So both the live-test pre-21:58 window AND the structural-duplicate analysis post-21:58 are valid — they're against the same hook code, just before/after the retry patch landed.
- **`belfry.log` is the authoritative timeline.** Every send path logs a distinctive line (`sent X: outbound reply …`, `sent X: ready …`, `brain reply …`, `dedup: skipped …`). When the user reports a duplicate symptom, grep on slug + send paths reconstructs the user's chat-side experience message-by-message. Worth keeping that log shape stable.
- **Reply-tracker is just `[message_id, slug]`** — no text. To reconstruct what a given outbound message *said*, you need the daemon log's char-count + the assistant transcript. Adequate for diagnosis; would be a feature gap if we ever wanted full audit-trail.
