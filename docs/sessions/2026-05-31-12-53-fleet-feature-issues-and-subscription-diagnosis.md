## Key Learnings
- **`notifications/claude/channel` content is wrapped in `<channel source="..." slug="..." ts="...">` tags BEFORE the model sees it.** Confirmed by reading `bin/belfry-mcp.js` `injectChannelMessage` and matching against the system reminder that documented the same tag shape. Implication: Claude Code's slash-command parser never sees channel-injected text. `/compress fast` would be literal text containing a slash; `compress fast` (natural language) works because the model interprets it. Material to #30: the broadcast contract is "inject text the model interprets," not "execute slash commands in N sessions."
- **`markOwesReply(slug, originalMessageId)` is the natural threading primitive for fleet replies.** Already in `lib/registry.js`. For a Telegram-originated `/all <text>` broadcast, calling it for every targeted slug at broadcast time makes every session's later `reply()` call quote-reply to the user's `/all` message automatically — N session responses thread under the broadcast in Telegram for free. Saved a tracker-rewrite proposal I'd have written without reading the code.
- **The bidirectional reply path is per-session MCP, completely independent of `belfry.jsonc`.** Realized while diagnosing cybersecurity: belfry-mcp's `reply` tool POSTs to the registry's `/send` endpoint, which calls `onSend` and emits to Telegram. Subscriptions in `belfry.jsonc` gate ONLY the daemon's `onUpdate` watcher → composer → `sendMessage` path for status pings. Two independent outbound flows; the whitelist applies to exactly one of them. Worth documenting more explicitly in `CLAUDE.md` and the README — this is a footgun for the user's mental model.
- **`$BELFRY_RESUME_LAUNCHER` is the right abstraction for any "drive a frontend from belfry" use case.** Detached subprocess, env vars in, daemon agnostic about what runs downstream. Generalizes cleanly to spawn (`BELFRY_SPAWN_LAUNCHER`), kill (`BELFRY_KILL_LAUNCHER`), and any future "tell the local environment to do something" surface. Keeps belfry from drifting into multiplexer territory.
- **Issue-as-design-conversation is the right pattern for v0.2.0-class scope.** #28 set the precedent (Motivation → Shape → Open Questions → Files this would touch → Why this is filed not implemented). #29, #30, #31 followed it. Future-me reading these can see the full design surface and the trade-offs that were live at filing time, not just "what we built." Counts as the design RFC without inventing a new doc format.
- **`secrets-manager` is the always-available auth source.** Reference memory now points to it; the pattern (pipe to env var of consuming command, don't persist) applies to any tool that errors with "please auth." Eliminates the "I'll ask the user to auth" failure mode I had before this session.
